# 多 AI 角色协作群聊系统 — 讨论结论总结

> 基于 OpenCode SDK/Server 的可行性研究与方案讨论，2026-05-11

---

## 1. 需求概述

构建一个类似聊天群的协作工具，包含人类和 AI 角色：

- 每个参与者有公司内的职能角色（架构师、运营、文案等）
- 围绕"项目"（不限于代码项目）进行工作
- 基于职责进行工作交接，最终完成项目
- 每个 AI 角色背后是一个独立的 OpenCode Serve 实例，通过 SDK 调用
- 同一群/语境对应不同 AI 的 session，做好关联
- AI 从群聊中感知上下文、明确任务、完成后交接
- 通过 Docker 部署各 AI 角色实例
- 相同角色模板（如文案），在不同公司实例化后有各自的公司/项目记忆

---

## 2. 可行性结论

| 能力 | 可行性 | 说明 |
|------|--------|------|
| Docker 部署 opencode serve | ✅ 完全可行 | 无界面 HTTP 服务器，支持端口/主机名/CORS/Basic Auth 配置 |
| SDK 调用控制 AI | ✅ 完全可行 | session CRUD、消息收发、事件订阅、结构化输出均支持 |
| 角色配置系统 | ✅ 完全可行 | 通过 opencode.json 的 Agent 配置 + Markdown agent 文件定义角色 |
| 工作交接 | ⚠️ 需自建 | 利用结构化输出定义 handoff 协议，编排层做路由 |
| 公司/项目记忆 | ⚠️ 需设计 | 文件系统 + 编排层数据库结合 |
| 群聊上下文感知 | ⚠️ 需自建 | 编排层写入文件 + SDK 通知触发 |

**核心结论**：OpenCode 提供了所有底层能力（AI 对话、session 管理、配置、上下文注入），编排逻辑（消息路由、交接协议、记忆管理、群-session 映射）需要自建。

---

## 3. 记忆架构：三层设计

### 最终方案：文件系统管长期记忆，Session 管短期工作上下文

```
Layer 1: 角色身份（Docker 镜像内，不可变）
├── opencode.json       — Agent 定义、工具权限、模型配置
├── role/capabilities.md — 角色能力描述
└── .opencode/          — skills、agents、plugins、commands

Layer 2: 公司/项目上下文（容器挂载，半静态）
├── AGENTS.md           — 公司信息 + 项目信息 + 经验目录索引
├── company/info.md     — 公司详情
└── project.md          — 当前项目描述

Layer 3: 动态经验（Volume 挂载，AI 可写）
├── rules/_meta.md      — 经验索引（自动加载，轻量）
├── rules/exp-xxx.md    — 详细经验文件（按需 lazy load）
└── context/            — 当前任务上下文（编排层写入）
```

### 关键决策

- **AGENTS.md** 进入 System Context（系统提示词层），只加载一次，token 效率高
- **rules/** 通过 opencode.json 的 `instructions` 通配加载（`rules/_meta.md`），详细文件由 AI 按需读取
- **context/** 由编排层写入当前群聊上下文，AI 被 prompt 通知后自行读取
- SDK 不再做内容注入，只做**通知触发**（"有新消息了，去读文件"）

### 被否决的方案

- **纯 SDK 注入**（每次 noReply 注入记忆）：每轮重复注入造成 token 大量浪费，对话历史膨胀
- **纯 AGENTS.md 静态方案**：无法动态更新，运行中修改需重启容器

## 4. 部署架构

### 角色模板 = Docker 镜像

```
角色模板（求职者）          公司聘请（入职）
┌──────────────────┐     ┌──────────────────────────┐
│ Docker 镜像       │     │ 运行中的容器               │
│ opencode.json    │     │ 挂载:                     │
│ role/            │ ───►│   AGENTS.md (公司+项目)    │
│ .opencode/       │ 挂载 │   company/ (公司信息)      │
│ (skills/plugins) │ 组合 │   rules/ (动态经验, volume) │
└──────────────────┘     │   context/ (临时上下文)     │
                         └──────────────────────────┘
```

- 一个角色 = 一个通用 Docker 镜像
- "聘请" = 启动容器 + 挂载公司/项目专属文件
- 同一角色服务 N 家公司 = N 个容器实例
- 动态经验通过 Docker Volume 持久化

## 5. Session 策略

### 结论：每个（群 × 角色）维持一个长驻 Session

| 策略 | 评价 |
|------|------|
| 每条消息新建 session | ❌ AI 丢失自己的思考链，如同失忆的无头模式 |
| 每群每角色长驻 session | ✅ 推荐，AI 保持工作上下文连贯 |
| 按（群 × 角色 × 任务阶段）session | ⚠️ 折中方案，增加编排复杂度 |

### Session 生命周期

```
群/任务启动 → 每个参与角色 session.create()
                ↓
任务进行中 → 同一 session 内多轮 prompt
             AI 累积自己的思考链和工作上下文
                ↓
session 过长（>20轮）→ 保存经验到 rules/ → summarize → 轮换新 session
                ↓
任务结束 → save-experience → session.delete()
```

### 关键点

- **不是无头模式**：AI 在同一 session 里有多轮连续的思考
- **膨胀控制**：session 超过阈值时轮换，经验写入文件保证连续性
- **并行安全**：同一角色的不同群使用不同 session

## 6. 群聊运行时机制

### 消息流转

```
1. 群内消息发出
2. 编排层判断路由（人类指定 / 流程驱动 / AI handoff）
3. 编排层写入 context/current-task.md（群聊上下文）
4. 目标角色的 opencode session.prompt() 触发通知
5. AI 读取 context 文件，完成工作，返回结果
6. 编排层解析结果（含结构化 handoff 信息）
7. 根据 handoff 路由给下一个角色或人类
8. 更新工作记要
```

### 工作交接协议

AI 完成工作后通过结构化输出返回交接信息：

```json
{
  "response": "群聊中的回复内容",
  "handoff": {
    "next": "运营",
    "reason": "文案已完成，需要运营制定投放策略",
    "context": "给下一个角色的补充说明"
  },
  "need_human": false
}
```

### 人类角色

- 审批门：AI 完成关键节点后暂停，等待人类审核
- 打断/重定向：人类可随时 `@角色` 改变工作方向
- 审批模式：同步等待 / 异步继续 / 代理自动审批

## 7. 项目与群聊的关系

**项目 ≠ 群聊，是 1:N 关系**

```
项目 "夏季推广"
  ├── 群聊1: 整体策划（架构师 + 运营 + 文案 + 人类）
  ├── 群聊2: 小红书文案专项（文案 + 运营 + 人类）
  └── 群聊3: 视频脚本讨论（文案 + 架构师 + 人类）

共享:
  ├── 工作记要（跨群聊的进度和历史）
  ├── 项目级 rules/（所有群的经验共享）
  └── deliverables/（产出物归档）
```

## 8. 完整生命周期

```
① 公司入驻        → 公司注册、信息录入、团队成员管理
② 配置角色模板     → 定义角色能力、skills、MCP、plugins
③ 聘请/生成员工    → 实例化角色容器，挂载公司信息
④ 创建项目         → 项目描述、目标、关联员工
⑤ 生成群聊         → 选择参与角色+人类，加载项目上下文
⑥ 群聊运行时       → 消息路由、工作交接、人类审批
⑦ 项目沉淀         → 产出归档、经验提取、工作记要
⑧ 员工/项目管理    → 记忆管理、角色升级、成本统计
```

---

## 9. 风险与注意事项

| 风险 | 应对策略 |
|------|---------|
| Token 消耗 | AGENTS.md 做轻量索引 + 按需读取详细文件 |
| 多角色串行延迟 | 可并行的任务同时触发多个角色 |
| OpenCode 为代码场景设计 | 非 code 项目需精心设计 prompt 和工作流 |
| Session 数据持久化 | Docker Volume 正确挂载 |
| 容器重建时记忆丢失 | rules/ 和 AGENTS.md 使用持久化 Volume |
| 角色模板升级 | 新镜像 + 旧 Volume，记忆不丢 |
